> 版本:v1.0
> 日期:2026-06-13
> 状态:设计阶段收口

---

## 一、项目概述

### 1.1 目标

打造一套**货代(Freight Forwarder)自动化作业系统**,减少人工在订舱、补料、跟踪、对账上的重复操作,核心是把货代日常 70% 的"催、抄、录、查"工作交给系统做。

### 1.2 核心特性

- **自动订舱**:订舱邮件自动发、自动催、放舱自动识别
- **自动补料**:提柜后条件齐了自动触发补料
- **全链路跟踪**:柜子从提柜到还空柜,看板一站看完
- **报关/清关**:出口报关 + 进口清关状态全程可追
- **智能识别**:接 OpenClaw + MiniMax M3,SO 邮件、账单 PDF 自动结构化
- **多端覆盖**:Web / Windows exe / macOS dmg
- **开放 API**:客户、报关行、国外代理可对接

### 1.3 业务范围

出口段:订舱 → 提柜 → 装柜 → 报关 → 补料 → 装船 → 离港
进口段:到港 → 清关 → 提货 → 派送 → 还空柜

## 二、模块设计

### 2.1 订舱板块

#### 2.1.1 业务说明

货代接收客户的订舱委托,选择船代/船东,生成订舱邮件发给对方,等待对方放舱(SO)。期间需要催单、识别对方回复。

#### 2.1.2 关键决策

| 项 | 决策 |
|---|------|
| 邮件通道 | 腾讯企业邮 IMAP |
| 收件人 | 具体业务员 + 通讯录维护 |
| 模板维护 | IT 写死 + 下拉选项 |
| 发送方式 | 预览后再发 + 附带订舱委托单 PDF |
| 催单节奏 | 2h 软催 / 4h 硬催 / 8h 弹窗 |
| 催单文案 | 软硬两套,分开维护 |
| 多单处理 | 一笔一封 |
| 同步频率 | 15 分钟 |

#### 2.1.3 订舱流程

```
客户提交订舱委托
    ↓
操作员在系统建单(选船代/港口/柜型/货好时间)
    ↓
系统匹配邮件模板 + 填变量
    ↓
生成预览(正文 + PDF 附件)
    ↓
操作员点确认 → 发送
    ↓
状态:[订舱已发送]
    ↓
T+2h 未读 → 软催邮件
    ↓
T+4h 未回 → 硬催邮件(抄送主管)
    ↓
T+8h 仍未回 → 弹窗提示,人工电话
    ↓
收到船代回复 → M3 意图分类
    ├── 放舱 → 走 SO 识别流程
    ├── 拒舱 → 走换船东流程
    ├── 改价/改时间 → 弹窗让人决策
    └── 已读不回 → 升级催单
```

### 2.2 SO 识别 + M3 板块

#### 2.2.1 业务说明

船代放舱后,会发回 SO 邮件(Booking Confirmation),里面包含船名航次、起运港、目的港、ETD、截关时间、柜型柜量等关键信息。M3 自动结构化这些信息,人工复核后落库。

#### 2.2.2 关键决策

| 项 | 决策 |
|---|------|
| 调用方式 | 分步(意图+关键字段 → 细字段) |
| 邮件解析 | 正文走 M3 文本 / PDF 走视觉识别 |
| PDF 页数 | 5 页内,整批送 |
| 字段校验 | 宽松,只标黄不阻断 |
| 复核界面 | 表格 + 原文联动,高亮定位 |
| 下游推送 | 内部状态 + 财务预估 |
| 置信度展示 | 每字段带置信度,低值标黄 |

#### 2.2.3 识别流程

```
船代回信到达专用收件箱
    ↓
IMAP 拉取(15 分钟轮询)
    ↓
预处理:提取发件人、提取正文 + 附件
    ↓
判断是否 SO 邮件(发件人 / 主题 / 关键词)
    ↓
非 SO → 走意图分类
    ↓
是 SO → M3 结构化抽取
    ↓
字段校验(宽松,只标黄)
    ↓
写入复核队列
    ↓
操作员在复核界面确认 / 修正
    ↓
确认 → 落库 + 触发下游(内部状态 + 财务预估)
```

#### 2.2.4 M3 Schema(SO 抽取)

```json
{
  "so_number": "BOOKING12345",
  "carrier": "MSK",
  "vessel": "MAERSK HONG KONG",
  "voyage": "FE123W",
  "pol": "CNSHA",
  "pod": "USLAX",
  "etd": "2026-06-20",
  "cut_off_si": "2026-06-18 18:00",
  "cut_off_cy": "2026-06-19 12:00",
  "cntr_allocation": [
    {"type": "40HQ", "qty": 2}
  ],
  "special_instructions": "冻柜 -18℃",
  "_confidence": {
    "so_number": 0.98,
    "etd": 0.95,
    "cut_off_si": 0.92
  }
}
```

#### 2.2.5 复核界面

- 左侧:PDF 原文 / 邮件正文
- 右侧:字段表单(每个字段含输入框 + 置信度)
- 字段在原文中的位置**高亮**
- 点击字段 → 原文跳转到对应位置

### 2.3 补料板块

#### 2.3.1 业务说明

放舱后,货代要把发货人、收货人、通知人、柜号、货描、HS Code 等信息整理成 SI(补料)发给船代。提柜之后才发,因为得有柜号。

#### 2.3.2 关键决策

| 项 | 决策 |
|---|------|
| 收发通来源 | 通讯录 + 下拉引用 + 手动新增 |
| 触发机制 | 后端事件触发(WebSocket) |
| 附件格式 | Excel |
| 模板维护 | IT 写死 |
| 船代回"确认" | 通知操作员 |
| 船代回"让改" | 重新走完整补料流程 |

#### 2.3.2 补料流程

```
提柜完成(柜号 + 封号 入库)
    ↓
等"收发通"信息齐(发货人 / 收货人 / 通知人)
    ↓
两个条件都齐 → 后端事件触发(WebSocket 推前端)
    ↓
补料表单自动亮起
    ↓
操作员确认/补充信息
    ↓
系统生成 SI Excel 附件
    ↓
预览 → 确认 → 发送
    ↓
收到船代回复
    ├── 确认 → 状态:SI 已提交,通知操作员
    └── 让改 → 重新走完整补料流程
```

### 2.4 柜子状态跟踪

#### 2.4.1 业务说明

柜子从出场到收货,全生命周期状态跟踪,看板视图,操作员一站看完。

#### 2.4.2 生命周期节点

```
提柜(空柜出场)
  ↓
装柜(货进柜、封箱)
  ↓
还柜 / 进港(重柜进码头)
  ↓
报关放行
  ↓
装船(离港 ETD)
  ↓
中转(如果需要)
  ↓
到港(ETA)
  ↓
卸船
  ↓
提柜(收货人提货)
  ↓
还空柜(空柜还回场站)
  ↓
完结
```

#### 2.4.3 关键决策

| 项 | 决策 |
|---|------|
| 数据来源 | 手动查船公司网站 + 第三方 API 抓取 |
| API 信任度 | 直接信,不复核 |
| 状态通知 | 站内信,只通知操作员 |
| 视图 | 看板分列(提柜中/在途/到港/派送中/已提货) |

#### 2.4.4 看板列

| 列名 | 说明 |
|------|------|
| 提柜中 | 提柜到装柜之间 |
| 在途 | 已进港/装船 |
| 到港 | 已到目的港 |
| 派送中 | 派送中(合并到派送模块) |
| 已提货 | 提货完成 |

### 2.5 报关板块(出口)

#### 2.5.1 业务说明

货物出口前,委托报关行向海关申报,获得放行后才能装船。

#### 2.5.2 关键决策

| 项 | 决策 |
|---|------|
| 数据源 | 报关行 API(优先) / 手动录入(兜底) |
| 状态 | 待报关 / 报关中 / 已放行 / 查验中 |
| API | 设计接口先留着,等对接 |

#### 2.5.3 状态说明

| 状态 | 含义 |
|------|------|
| 待报关 | 单据未提交 |
| 报关中 | 海关审核中 |
| 已放行 | 审核通过 |
| 查验中 | 海关查验,需配合 |

### 2.6 清关板块(进口)

#### 2.6.1 业务说明

货物到国外港口后,由国外代理协助收货人完成清关。我们作为中国端货代,负责生成清关资料推给国外代理,并接收他们回推的状态。

#### 2.6.2 关键决策

| 项 | 决策 |
|---|------|
| 业务方向 | 进口清关(国外代理) |
| 我们的角色 | 推送方(推单据资料) |
| 状态来源 | 国外代理回推 |
| 状态细分 | 单据提交/海关审核/查验/放行/提货 |
| 鉴权 | API Key(跟报关一致) |
| 通知 | 站内信通知操作员 |
| 资料生成 | 操作员手动填,不用 M3 |

#### 2.6.3 流程

```
我们系统生成清关资料(发票/箱单/产地证等)
    ↓
推送给国外代理
    ↓
国外代理拿去做清关
    ↓
清关状态回推给我们(单据提交/海关审核/查验/放行/提货)
    ↓
通知操作员
```

### 2.7 装载状态

#### 2.7.1 业务说明

实时同步柜子的装载状态,从船公司的角度能看到柜子排上哪条船、是否装船。

#### 2.7.2 关键决策

| 项 | 决策 |
|---|------|
| 数据源 | 爬虫 / 第三方 API |
| 颗粒度 | 排载 + 装船 |
| 通知 | 通知操作员(站内信) |

#### 2.7.3 状态

| 状态 | 含义 |
|------|------|
| 已排载 | 船公司已确认柜子上船 |
| 已装船 | 船已离港 |

### 2.8 派送板块

#### 2.8.1 业务说明

柜子到港、提货之后,通过拖车派送到收货人仓库。我们只跟踪,不自己派车。

#### 2.8.2 关键决策

| 项 | 决策 |
|---|------|
| 业务角色 | 只跟踪,不自己派车 |
| 数据来源 | 派送 API / 收货人反馈 / GPS 三路 |
| 状态细分 | 派车/提货/运输中/已送达/异常 |
| 通知 | 不通知,看板自看 |
| 看板 | 合并到柜子状态看板 |

#### 2.8.3 状态

| 状态 | 含义 |
|------|------|
| 派车 | 已派出拖车 |
| 提货 | 已从港口提货 |
| 运输中 | 在去收货人仓库的路上 |
| 已送达 | 已送到 |
| 异常 | 改派/丢件/事故 |

### 2.9 财务板块

#### 2.9.1 业务说明

账单识别、入账、海运费预估展示。账单来源主要是船公司、拖车公司、报关行。

#### 2.9.2 关键决策

| 项 | 决策 |
|---|------|
| 账单输入 | PDF / Excel / 扫描件 三路 |
| 复核 | 强制人工复核再入账 |
| 匹配方式 | SO/提单号 / 柜号 都要支持 |
| 海运费来源 | 系统固定价(预估) + 账单识别(最终) |
| 状态展示 | 预估/已对账 标 |
| 差异提醒 | > 5% 标红 |

#### 2.9.3 账单识别管线

```
账单文件
    ├── PDF → 视觉识别(M3)
    ├── Excel → 表格解析(直接读 cell)
    └── 扫描件 → OCR + M3
    ↓
输出结构化 JSON
    ↓
写入复核队列
    ↓
操作员复核/修正
    ↓
入账
```

#### 2.9.4 海运费展示

- SO 状态栏同时显示**预估**和**最终**两列
- 预估 = 系统固定价
- 最终 = 账单识别回来覆盖
- 差异 > 5% 标红

### 2.10 数据互通

#### 2.10.1 业务说明

模块间数据要互通,看板统一,字段联动,避免重复录入和数据不一致。

#### 2.10.2 关键决策

| 项 | 决策 |
|---|------|
| 互通程度 | 看板统一(A) + 字段联动(B) |
| 联动方向 | 单向(源头改 → 下游跟) |
| 主数据 | SO/提单号(订舱定)、柜号(提柜定)、收发通(补料定) |
| 源头改下游 | 弹窗确认后同步 |
| SO = 提单号 | 统一字段 |

#### 2.10.3 主数据规则

| 字段 | 源头模块 | 谁可改 |
|------|---------|--------|
| SO/提单号 | 订舱 | 订舱模块(其他模块只读) |
| 柜号 | 提柜 | 提柜模块(其他模块只读) |
| 收发通 | 补料 | 补料模块(其他模块只读) |

源头字段修改时:**弹窗提示"将同步给下游模块,是否继续?"**,操作员确认后同步。

### 3.2 技术选型

| 项 | 选型 | 理由 |
|---|------|------|
| OpenClaw 位置 | 同服务器,内网 RPC | 低延迟、高安全 |
| 客户端框架 | Tauri | 跨平台一套代码,包小启动快 |
| 数据库 | PostgreSQL | 成熟稳定,支持 JSONB 灵活字段 |
| 缓存/队列 | Redis | 性能高,Streams 适合消息流 |
| 文件存储 | 本地磁盘起步 | MVP 简化,后期迁 OSS |
| 部署 | Docker Compose | 一键起,运维友好 |
| 智能体 | OpenClaw 编排 + M3 调用 | 已部署,直接接入 |

### 3.3 客户端分布

| 平台 | 形态 | 备注 |
|------|------|------|
| Web | 浏览器 | 主入口,无需安装 |
| Windows | Tauri 打包 .exe | 桌面快捷启动 |
| macOS | Tauri 打包 .dmg | 桌面快捷启动 |

三个平台共用一套 Web 前端代码,Tauri 跨平台编译。

## 四、API 设计

### 4.1 调用方

| 调用方 | 用途 | 鉴权 |
|--------|------|------|
| 客户 | 下单、查状态、补料推送 | API Key |
| 报关行 | 推报关状态 | API Key |
| 国外代理 | 推清关状态 | API Key |

### 4.2 鉴权方式

所有外部 API 使用 **API Key** 鉴权:

```
Authorization: Bearer <api_key>
X-Request-Timestamp: <unix_timestamp>
```

### 4.3 文档形式

静态 Markdown 文档,放在 `/docs/api/` 目录,随代码仓库版本管理。

### 4.4 计划接口列表(初稿)

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 创建订舱单 | POST | /api/v1/bookings | 客户下单 |
| 查询订舱单 | GET | /api/v1/bookings/{so} | 查详情 |
| 推送补料 | POST | /api/v1/bookings/{so}/si | 客户推 SI |
| 推送报关状态 | POST | /api/v1/bookings/{so}/customs | 报关行推 |
| 推送清关状态 | POST | /api/v1/bookings/{so}/clearance | 国外代理推 |
| 推送装载状态 | POST | /api/v1/bookings/{so}/loading | 装载/船公司 |
| 推送派送状态 | POST | /api/v1/bookings/{so}/delivery | 派送公司推 |

> 详细字段、错误码、限流规则在 API 文档中定义。

## 六、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| M3 识别准确率不足 | SO 数据错误 | 强制人工复核 + 置信度提示 |
| 船公司反爬 | 装载数据抓不到 | 切换第三方 API,或购买数据源 |
| 报关行 API 滞后 | 状态不实时 | 手动录入兜底 |
| OpenClaw 故障 | 整个识别挂掉 | 消息重试 + 死信队列 |
| 柜子状态数据孤岛 | 客户投诉 | 看板统一,数据互通 |
